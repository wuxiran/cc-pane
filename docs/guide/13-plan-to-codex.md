# 13. Plan → Codex 交接 & Plan 同行评审

> 两个玩法，共同点是**让 Claude 负责"想"、让 Codex 负责"做"或"挑刺"**——把规划和执行/评审分给不同的 AI，各司其职。

## 玩法一：Plan → Codex 交接（Claude 规划，Codex 实现）

**是什么**：Claude 把方案写成一份 plan 文件，自己**不**改代码，而是把 plan 交给一个 Codex 实例去落地。
好处：Claude 专注规划、不被实现细节拖累；Codex 在干净的新窗口里专心写代码；两边 context 互不挤占。

**什么时候用**：
- plan 已经比较成熟，就等执行
- 实现工作量大（不是几十行的小补丁）
- 想让 Codex 在另一个窗口 / WSL 里跑，不占用当前 Claude

**怎么发起**：对 Claude 说“把这个 plan 派给 Codex 实现”“plan to codex”。

**流程**：
1. Claude 写好 plan（通常在 plan 模式里）。
2. Claude 问你：**Codex 在哪跑？**（本地新窗口 / WSL / 复用已有窗口）
3. 开一个 Codex，把 plan 交给它执行。
4. Codex 干完**自动回报**，Claude 给你汇报：改了哪些文件、有没有跑测试。
5. 你决定下一步（跑测试、审 diff、合并）。

## 玩法二：Plan 同行评审（另一个 Codex 只审不改）

**是什么**：Claude 写完初版 plan 后，另开一个 Codex **只评审、不改代码**，从多个维度给方案挑毛病。
解决的是 **“自己审自己”的盲区**——换一个 AI 的视角，常能发现“这个 UI 控件根本不存在”“这步数据迁移没写回滚”“跨服务时序有问题”等你和 Claude 都没注意到的坑。

**什么时候用**：
- plan 比较大、涉及高风险（UI 端到端、数据迁移 / 回滚、跨服务时序、线上变更）
- 你想在动手前多一道把关

**怎么发起**：写完 plan 后对 Claude 说“找 Codex 审一下这个 plan”“peer review”。

**Codex 评审的 6 个维度**：
1. **业务时序**——步骤先后对不对，有没有并发 / 竞态
2. **数据耦合**——跨表 / 跨服务的数据依赖处理全了吗
3. **UI 可行性**——方案里说的按钮 / 页面是不是真存在、能点
4. **数据红线**——生产数据、隐私字段、不可逆操作的边界
5. **回滚完整性**——每一步出错能不能回滚、能否恢复原状
6. **未覆盖场景**——异常路径、边界值、不同权限 / 角色

**流程**：
1. Claude 写好初版 plan。
2. 开一个 Codex 评审，它给出**三类结论**：✅ 稳妥 / ⚠️ 必须改 / ❓ 需你拍板。
3. Claude 把“必须改”和“需拍板”的条目**逐条**摆给你，每条让你选 **采纳 / 改一下再采纳 / 拒绝**。
4. 按你的决定，Claude **整体重写** plan（不是打补丁）。

## 两个玩法怎么选 / 怎么配合

| | Plan → Codex 交接 | Plan 同行评审 |
| --- | --- | --- |
| Codex 干嘛 | **执行**（改代码） | **评审**（不改代码） |
| 评审后 | 直接出实现 | Claude 重写 plan |
| 你要拍板吗 | 基本不用 | 要，逐条拍板 |

**推荐配合**：高风险任务先**评审**（玩法二）→ 你拍板 → 重写 plan → 再**派 Codex 实现**（玩法一）。

## 注意

- **plan 要先写好**，空手派 Codex 效果差。
- 在 **WSL** 里跑 Codex 时，你在对话里贴的 Windows 路径要转成 Linux 形式（见 [15. WSL / SSH 远程运行](15-wsl-ssh.md) 的路径转换表）。
