# 11. 多实例并行跑任务

> 一个大任务，拆成几块互不相干的小任务，开几个 AI 实例**同时**做——而不是排队一个个来。这是 CC-Panes [MCP 编排能力](mcp-orchestration.md) 最直接的用法。

## 什么时候用、什么时候别用

**该用**：
- 任务能拆成 2 个以上**互不依赖**的子任务（改的不是同一批文件、没有先后顺序）
- 每块都够大，并行能明显省时间（大重构、多个独立模块、批量检查）

**别用**：
- 子任务共享同一文件，或后一步依赖前一步 → 老老实实串行，更省心
- 小修小补（几十行）→ 并行的开销不值得

## 怎么发起

对 Claude 说一句就行，比如：

> “这几件事互相独立，能并行跑吗？”
> “分头执行，开几个 worker 同时做。”

Claude 识别后会先给你一份**拆分方案**，你确认了它才动手。

## 背后发生了什么（大致了解即可）

1. Claude 把任务拆成几个独立子任务，**先给你确认**。
2. 对每个要**改代码**的子任务，CC-Panes 给它开一个独立的 **Git worktree**——相当于给每个 worker 一份独立的工作副本，彼此不会写串、不会撞 Git。
3. 只读的活（研究、检查、审查）不占 worktree，用更轻量的方式跑。
4. 几个 worker 并行干活，Claude 盯着进度。
5. 全部完成后，Claude 汇总每个 worker 改了什么、给出 diff。

## 你要做什么

- **确认拆分方案**（是否合理、要不要调整）
- 想看细节就切到各个 worker 的标签页看实时输出
- 看最后的**汇总表**，确认合并顺序——**Claude 不会自动提交，等你点头**

## 注意

- **子任务必须真的独立**：一旦有依赖或共享文件，并行就会打架，不如串行。
- **代码型 worker 一定走 worktree 隔离**（CC-Panes 自动处理），这是并行不冲突的关键。
- **合并由你拍板**：AI 拉 diff，但不替你 commit。
